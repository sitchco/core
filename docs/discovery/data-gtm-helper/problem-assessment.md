# Problem Assessment: `data-gtm` Attribute Helper & Auto-Injection

## Summary

The new platform needs a way to place `data-gtm` attributes in Twig templates for GTM click tracking, with zero cost when TagManager is disabled. The old platform solved this with a global PHP function and hardcoded auto-injection in the Backstage component framework — neither of which exists in the new architecture. The new platform's module system, Twig function registration, and `block_attributes` convention provide cleaner extension points that the old platform lacked, but no single mechanism covers both blocks and structural elements.

## Twig Extension Points

All custom Twig functions are registered in `modules/TimberModule.php:33-50` via the `timber/twig/functions` filter. This is the only Twig extension mechanism in use — no `AbstractExtension`, no `timber/twig/filters`, no raw `timber/twig` environment hook. Adding a new function follows the established pattern:

```php
add_filter('timber/twig/functions', function ($functions) {
    $functions['gtm_attr'] = [
        'callable' => ...,
        'is_safe' => ['html'],
    ];
    return $functions;
});
```

The `is_safe => ['html']` option prevents Twig from escaping the return value, which is necessary since the function would return raw HTML attribute markup.

There is no existing example of an optional Twig function — one that exists only when a specific module is active. All current Twig functions are registered by `TimberModule`, which is always active. A TagManager-specific function registered only by TagManager would cause templates to fail when TagManager is disabled, unless a no-op fallback is registered by a base module.

**Key file:** `modules/TimberModule.php:33-50`

## Block Rendering Pipeline

Every Timber/ACF block renders through `TimberModule::blockRenderCallback` (`modules/TimberModule.php:80-120`):

```
register_block_type (render_callback)
  → setupContext()           — Timber context + ACF fields
  → loadBlockScope()         — include block.php (parent theme, then child theme)
  → mergeMetadataContext()
  → normalizeInnerBlocksContext()
  → TimberUtil::compileWithContext('block.twig', $context, "block/$blockName")
      → apply_filters('sitchco/template-context/block/{blockName}', $context)
      → Timber::compile($template, $context)
  → Block::wrapperElement()  — wraps with get_block_wrapper_attributes() (outer div)
```

This produces **two nested wrapper layers**:

1. **Outer WordPress wrapper** — generated by `Block::wrapperElement()` → `get_block_wrapper_attributes()`. This is the `<div class="wp-block-...">` that WordPress adds. Attributes can be injected here by passing them into `get_block_wrapper_attributes()`.

2. **Inner template root element** — the Twig template's own outermost element, which uses `{{ block_attributes }}` if the block follows the convention.

For blocks like `featured-link`, the outer wrapper is a `<li>` and the inner element is an `<a>` — the semantically meaningful element is Layer 2, not Layer 1.

**Key files:** `modules/TimberModule.php:80-120`, `src/Utils/Block.php:33-46`

## The `block_attributes` Convention

Multiple blocks follow a convention where `block.php` builds an attribute array, converts it to an HTML string, and passes it to Twig:

```php
// block.php
$block_attributes['class'] = ['featured-link'];
$block_attributes['href'] = $fields['url'] ?? null;
$context['block_attributes'] = ArrayUtil::toAttributes($block_attributes);
```

```twig
{# block.twig #}
<a {{ block_attributes }}>
```

`ArrayUtil::toAttributes()` (`src/Utils/ArrayUtil.php:103-124`) handles zero-cost natively: `false` and `null` values are silently dropped from the output. A `data-gtm` key set to `null` produces no markup at all. This is a ready-made zero-cost pattern for blocks — but it's a convention, not a requirement. Some blocks (e.g., `content-slider`) bypass `{{ block_attributes }}` entirely and build their own root element attributes directly in Twig.

**Key files:** `src/Utils/ArrayUtil.php:103-124`, `roundabout/modules/Theme/blocks/featured-link/block.php:31-38`

## Structural Elements

The parent theme defines all structural landmarks:

- `<header>` in `views/partials/site-header-content.twig:6`
- `<footer>` in `views/partials/site-footer.twig:1`
- `<main id="main">` in `views/layouts/base.twig:17`

None of these elements have variable attribute slots. There is no mechanism to inject arbitrary attributes from outside the template. The child theme has no `views/` directory and cannot reach these elements without overriding entire Twig blocks.

To add `data-gtm` to structural elements, the parent theme templates must be modified to call a helper function:

```twig
<header class="..." {{ gtm_attr('Header') }}>
```

This is the same approach the old platform used — structural labels like "Header" and "Footer" were hardcoded directly in theme templates (`header.php:9`, `filter-group.php:11`). The difference is that the new platform can use a Twig function that returns empty when TagManager is disabled, whereas the old platform always rendered the attributes.

**Key files:** `sitchco-parent-theme/views/layouts/base.twig`, `sitchco-parent-theme/views/partials/site-header-content.twig`, `sitchco-parent-theme/views/partials/site-footer.twig`

## Zero-Cost Module Patterns

The module system guarantees that disabled modules register zero hooks: they are never instantiated, so their `init()` never runs and no `add_action`/`add_filter` calls execute. This is structurally enforced — no static initializers, no file-level hook registration.

The canonical zero-cost pattern is `Image::resize()` / `CloudinaryModule`:

```php
// src/Model/Image.php:132 — call site with safe default
$resized = apply_filters(Hooks::name('image/resize'), false, [...]);
if (false !== $resized) { return $resized; }
return ImageHelper::resize(...);  // fallback

// modules/Cloudinary/CloudinaryModule.php:22 — provider (only when active)
add_filter('sitchco/image/resize', [$this, 'imageResize'], 10, 2);
```

When Cloudinary is disabled, `apply_filters()` returns the default, and the fallback runs. Zero overhead from the absent module.

For a Twig function, the analogous approach is: register a no-op stub in `TimberModule` (always active), and let TagManager replace the callable when active. This avoids the "undefined function" problem while maintaining zero markup cost.

**Key files:** `src/Model/Image.php:132`, `modules/Cloudinary/CloudinaryModule.php:22`, `src/Framework/ModuleRegistry.php:102`

## Post-Render Injection via `render_block`

`ExtendBlockModule` demonstrates a post-render attribute injection pattern using the `render_block` WordPress filter and regex targeting the first HTML element (`ExtendBlockModule.php:38-99`). This is the only approach that works universally across all block types — including WordPress core blocks that have no `block.php` or `block_attributes` convention.

The tradeoff: post-render injection always targets Layer 1 (the outermost element), which is the WordPress wrapper `<div>`, not the template's semantic root element. For many blocks this is fine; for blocks like `featured-link` where the meaningful element is nested, it would attribute-tag the wrong element.

`WP_HTML_Tag_Processor` (used in `KadenceBlocks.php`) is a safer, regex-free alternative for post-render HTML attribute manipulation.

**Key file:** `modules/Wordpress/ExtendBlockModule.php:38-99`

## Template Context Injection

Every Twig compile passes through `TimberUtil::compileWithContext()` (`src/Utils/TimberUtil.php:16`), which fires `sitchco/template-context/{key}` before rendering. Any module can inject variables into a template's context via this filter. The convenience method `TimberUtil::addContext()` simplifies this.

For blocks: `sitchco/template-context/block/{blockName}` fires per block render.
For partials: `sitchco/template-context/partials/{name}` fires per partial.

A TagManager module could hook these filters to inject a `gtm_attr` context variable. When TagManager is inactive, the filter never registers, and the variable is absent. Templates referencing an absent variable need a default (e.g., `{{ gtm_attr|default('') }}`).

**Key files:** `src/Utils/TimberUtil.php:16-31`

## Old Platform Behavior

The old platform's `sd_gtm_attr()` was a global PHP function loaded unconditionally via Composer `files` autoload. It accepted strings, arrays, or objects — arrays/objects were `json_encode()`d, then everything was `esc_attr()`d. It returned either a full attribute string (`data-gtm='...'`, single-quoted) or just the escaped value.

Auto-injection happened through `Util::componentAttributesArray()` in the Backstage framework, which hardcoded `data-gtm => $base` (the component's BEM name) as the first key for every component. Explicit overrides via `elementAttributes()` won through `array_merge`.

There was no zero-cost behavior — `data-gtm` attributes were always rendered in HTML regardless of GTM status. The GTM container script tag was the only thing gated by the enable check.

**Key files (old platform):** `tag-manager/php/functions.php:9-17`, `backstage/src/Util.php:762-778`

## Resolved Questions

1. **What structure does the JS click tracker expect from `data-gtm` values?** Both plain strings and JSON objects are supported (defined in scenario spec Axiom 6). String values = context labels for DOM ancestry walk. JSON object values = bare-key interaction overrides. `"0"` or `"false"` = opt-out for the element. The `gtm_attr()` Twig function handles encoding: strings pass through, arrays/objects are JSON-encoded, all values are escaped for attribute output.

2. **Auto-injection: not pursued.** Auto-injecting `data-gtm` on every block root element adds noise without clear analytics value. Generic labels like `"wp-block-theme-featured-link"` don't provide meaningful context beyond what structural labels (Header, Footer) already provide. Instead, `data-gtm` placement is always explicit — theme developers add it where the context is analytically meaningful.

3. **Which elements get `data-gtm` labels?** Three tiers of explicit placement:
   - **Structural landmarks** (always): `Header`, `Footer`, `Main` — placed in parent theme layout templates
   - **Block templates** (system-level decision): A block template either includes a `data-gtm` context label or it doesn't — every instance of that block gets the same treatment. Not needed at launch since click tracking works without markup. Add to specific block templates (e.g., `Slider`, `Tabs`) if/when analytics data shows the context is valuable.
   - **Interaction overrides** (as needed): JSON objects on specific clickable elements to override auto-resolved labels

4. **Structural labels are theme-owned.** The parent theme templates must be modified to add `{{ gtm_attr('Header') }}` etc. No other mechanism can reach these elements. This matches the old platform pattern where structural labels were hardcoded in theme templates.

## Recommended Approach

**A Twig function with a no-op stub.** Register `gtm_attr` in `TimberModule` returning `''` by default. TagManager, when active, replaces the callable (via an `apply_filters` on the callable itself, or by hooking `timber/twig/functions` at a higher priority). This achieves:

- Zero markup when TagManager is disabled (function returns empty string)
- No undefined-function errors in templates
- Works in both block templates and structural partials
- Follows existing patterns (`timber/twig/functions` registration, Cloudinary-style opt-in enhancement)
- Theme templates express structural labels (`{{ gtm_attr('Header') }}`), TagManager provides the rendering logic

For blocks using the `block_attributes` convention, TagManager can additionally hook `sitchco/template-context/block/{name}` to inject `data-gtm` into the `$block_attributes` array before `ArrayUtil::toAttributes()` serializes it. This targets Layer 2 (the semantic root element) and requires no template changes. For blocks that don't follow the convention, the Twig function provides a fallback.
